# 第05章：推荐下注与执行路线

## 要回答的管理问题

管理层现在应批准哪一个战略下注，如何用阶段性证据决定继续、调整或停止？

## 输出结构

### 5.1 单一推荐

- 用一句话写清推荐：在哪个具体赛道，以哪个首发产品，为谁、解决什么、从哪个入口进入、通过什么独特方式、获得什么商业结果。
- 说明为什么推荐该选项，以及为什么不选第 4 章的其他选项。
- 明确该推荐依赖的 3 个关键假设。

### 5.3 指标与商业闭环

同时定义：

- 领先指标：需求、使用、质量或交付信号。
- 结果指标：收入、毛利、成本、留存、转化或风险指标。
- 战略指标：数据闭环、渠道控制、生态或组织能力是否形成。

没有可靠基础值时只给指标定义和测量方法，不捏造目标数字。

指标必须继承第 2 章识别出的收入引擎和最小经济单元。广告、订阅、交易、SaaS、API、IP/内容等模型使用各自的领先指标、贡献利润和盈亏平衡阈值，不得统一套用同一组增长指标。

### 5.4 Continue / Pivot / Kill 标准

给出每个决策门的继续、调整和停止条件，标明最迟决策时间与责任人类型。条件至少覆盖第 1 章市场规模敏感变量和第 2 章单位经济关键阈值。

### 5.5 风险与管理层待决事项

列出前五项风险、缓释动作，以及需要管理层本次批准的预算、组织、合作或风险偏好事项。

Deep 模式或启用 `stress_test` 时，至少增加一个行业相关压力情景：明确触发条件、业务影响、领先信号、响应动作和对应 Continue/Pivot/Kill 门槛。不得只写泛化的“市场不及预期”。

## 可选附录

仅当用户明确要求产品方案时，再输出 1–2 个产品概念或原型；不得让产品玩法替代战略选择。

## 写作要求

- 只给一个首选下注，不输出 4–5 个同优先级“玩法”。
- 建议必须能被证伪，资源、指标与停止条件缺一不可。
- 推荐只能依赖 `supported` 的关键结论；`mixed`、`provisional` 或 `inconclusive` 只能形成条件性建议或验证任务。
- 关键事实附 evidence_id 与 URL；推断和建议要与事实分开。
